## Overview

A filter is a callable that receives `(client, update)` and returns a boolean. When a filter returns `True`, the handler runs. When it returns `False`, the handler is skipped.

Filters decouple routing logic from handler logic. You define what kind of update you care about in a filter, then attach that filter to one or more handlers.

---

## `Filter` class

```python
from pytdbot.filters import Filter

my_filter = Filter(func=my_async_func)
```

`Filter` wraps a callable and makes it usable as a decorator or as a value passed to `filters` parameter on any handler decorator.

The wrapped callable must accept two positional arguments:

- `client` — the `Client` instance
- `update` — the TDLib update object (or inner object if `inner_object=True` on the handler)

It must return `True` to allow the handler to run, or `False` to skip it.

<ParamField path="func" type="Callable" required>
  A callable with signature `(client: Client, update) -> bool`. May be sync or async.
</ParamField>

Passing a non-callable raises `TypeError`.

---

## `filters.create()` factory

The recommended way to build a filter. Wraps a callable in a `Filter` instance.

```python
from pytdbot import filters
```

<ParamField path="func" type="Callable" required>
  The filter function. Must accept `(client, update)` and return `bool`. May be sync or async.
</ParamField>

Returns a `Filter` instance.

### Creation patterns

**As a decorator**

```python
from pytdbot import filters

@filters.create
async def filter_photo(_, event) -> bool:
    if event.content_type == "messagePhoto":
        return True
    return False
```

**As a function call**

```python
async def filter_photo(_, event) -> bool:
    return event.content_type == "messagePhoto"

filter_photo = filters.create(filter_photo)
```

**As a lambda**

```python
from pytdbot import filters

filter_photo = filters.create(
    lambda _, event: event.content_type == "messagePhoto"
)
```

## Using a filter with a handler

Pass a filter to the `filters` parameter of any handler decorator:

```python
from pytdbot import filters, Client, types

client = Client(...)

filter_photo = filters.create(
    lambda _, event: event.content_type == "messagePhoto"
)

@client.on_updateNewMessage(filters=filter_photo)
async def photo_handler(c, update):
    await update.message.reply_text("I got a photo!")

client.run()
```

---

## Using a filter as a decorator (plugin pattern)

When writing plugin modules, you can apply a filter directly as a decorator on your handler function. Pytdbot detects the `_handler` attribute that the filter decorator sets on the function and registers the handler when the plugin loads.

```python
from pytdbot import filters
from pytdbot.handlers import Handler

@filters.create
async def is_text(_, event) -> bool:
    return hasattr(event, "content") and event.content.getType() == "messageText"

@is_text
async def handle_text(client, message):
    await message.reply_text("Got a text message!")
```

When `is_text` is used as a decorator, it assigns `handle_text._handler` — a `Handler` instance with `update_type="updateNewMessage"` and the filter attached. Pytdbot's plugin loader reads this attribute during plugin registration.

## Complete example — filter for text messages only

```python
from pytdbot import filters, Client, types

client = Client(
    token="YOUR_BOT_TOKEN",
    api_id=0,
    api_hash="YOUR_API_HASH",
    files_directory="BotDB",
    database_encryption_key="changeme",
)


@filters.create
async def is_text_message(_, message) -> bool:
    """Accept only messages whose content is MessageText."""
    return (
        hasattr(message, "content")
        and message.content is not None
        and message.content.getType() == "messageText"
    )


@client.on_message(filters=is_text_message)
async def echo_text(client, message):
    await message.reply_text(message.text)


client.run()
```

<Tip>
  The first argument to a filter function is the `Client` instance. The convention `_` signals that it is intentionally unused. Use the argument when you need to make API calls inside the filter itself.
</Tip>
