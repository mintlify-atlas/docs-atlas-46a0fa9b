---
title: "Scaling with TDLib Server"
description: "Run multiple Pytdbot instances sharing a single TDLib connection using RabbitMQ and TDLib Server."
---

A single Pytdbot process handles one TDLib instance. When you need to run multiple worker processes — for example, to distribute update handling across machines — you can use [TDLib Server](https://github.com/pytdbot/tdlib-server) together with RabbitMQ.

## Architecture

```
Telegram ──► TDLib Server ──► RabbitMQ ──► Worker 1 (Pytdbot)
                                       ──► Worker 2 (Pytdbot)
                                       ──► Worker N (Pytdbot)
```

- **TDLib Server** holds the single authenticated TDLib connection and publishes updates to RabbitMQ.
- **RabbitMQ** distributes updates and relays API requests between workers and TDLib Server.
- **Pytdbot workers** connect to RabbitMQ instead of TDLib directly. They receive updates and send requests through the broker.

<Note>
  TDLib Server is a separate process. See [github.com/pytdbot/tdlib-server](https://github.com/pytdbot/tdlib-server) for installation and configuration instructions.
</Note>

## Connecting a worker to RabbitMQ

Pass `rabbitmq_url` to `Client`. Do not pass `lib_path` — the worker communicates through RabbitMQ, not a local TDLib binary.

```python
from pytdbot import Client, types

client = Client(
    token="BOT_TOKEN",
    api_id=0,
    api_hash="API_HASH",
    rabbitmq_url="amqp://guest:guest@localhost/",
    workers=5,
)

client.run()
```

## Instance ID

Each worker that connects to the same RabbitMQ broker needs a unique `instance_id`. If you omit it, Pytdbot generates a random 4-character ID at startup.

```python
client = Client(
    token="BOT_TOKEN",
    api_id=0,
    api_hash="API_HASH",
    rabbitmq_url="amqp://guest:guest@localhost/",
    instance_id="worker-1",  # must be unique per worker
)
```

<Tip>
  Use a stable, unique value per process (e.g. a hostname or container name) so RabbitMQ queues are consistent across restarts.
</Tip>

## Workers that don't receive updates

A worker that only sends API requests — for example, a scheduled-task runner — can set `no_updates=True` to skip update consumption entirely:

```python
sender = Client(
    token="BOT_TOKEN",
    api_id=0,
    api_hash="API_HASH",
    rabbitmq_url="amqp://guest:guest@localhost/",
    instance_id="sender",
    no_updates=True,
)
```

## Managing multiple clients in one process

`ClientManager` lets you run several `Client` instances inside a single Python process, sharing one TDLib receiver thread. This is useful for multi-account bots or testing.

```python
import asyncio
from pytdbot import Client, ClientManager

bot_a = Client(
    token="TOKEN_A",
    api_id=0,
    api_hash="API_HASH",
    database_encryption_key="key_a",
    files_directory="BotA",
)

bot_b = Client(
    token="TOKEN_B",
    api_id=0,
    api_hash="API_HASH",
    database_encryption_key="key_b",
    files_directory="BotB",
)

async def main():
    # Pass clients as a list — manager.start() registers and starts each one
    manager = ClientManager(clients=[bot_a, bot_b])
    await manager.start()

    # Both clients are now running and sharing one TDLib receiver thread
    await asyncio.gather(bot_a.idle(), bot_b.idle())

asyncio.run(main())
```

You can also add clients dynamically after the manager has started:

```python
await manager.add_client(bot_c, start_client=True)
```

And remove them:

```python
await manager.delete_client(bot_b.client_id, close_client=True)
```

### Shutting down

```python
await manager.close(close_all_clients=True)
```

## Server stats

When connected via RabbitMQ, call `getServerStats()` to retrieve runtime statistics from TDLib Server:

```python
stats = await client.getServerStats()
if not isinstance(stats, types.Error):
    print(f"Uptime:    {stats.uptime}s")
    print(f"Updates:   {stats.updates_count}")
    print(f"Requests:  {stats.requests_count}")
    print(f"My ID:     {stats.my_id}")
```

`ServerStats` fields:

| Field            | Type  | Description                           |
| ---------------- | ----- | ------------------------------------- |
| `my_id`          | `int` | Telegram user ID of the connected account |
| `uptime`         | `int` | Server uptime in seconds              |
| `updates_count`  | `int` | Total updates received by TDLib Server |
| `requests_count` | `int` | Total requests forwarded to TDLib     |

## Scheduled events

TDLib Server supports scheduling payloads for future delivery. Workers receive scheduled events as `updateScheduledEvent` updates.

### Schedule an event

```python
import time

event = await client.scheduleEvent(
    name="send_reminder",
    payload='{"chat_id": 123456789, "text": "Reminder!"}',
    send_at=int(time.time()) + 3600,  # 1 hour from now
)

if not isinstance(event, types.Error):
    print(f"Scheduled event ID: {event.event_id}")
    print(f"Will fire at:       {event.send_at}")
```

### Handle the event

```python
import json

@client.on_updateScheduledEvent()
async def on_scheduled(c: Client, update: types.UpdateScheduledEvent):
    if update.name == "send_reminder":
        data = json.loads(update.payload)
        await c.sendTextMessage(
            chat_id=data["chat_id"],
            text=data["text"],
        )
```

### Cancel an event

```python
result = await client.cancelScheduledEvent(event_id=event.event_id)
```

`ScheduledEvent` fields:

| Field      | Type  | Description                              |
| ---------- | ----- | ---------------------------------------- |
| `event_id` | `int` | Unique identifier of the scheduled event |
| `send_at`  | `int` | Unix timestamp when the event fires      |

`UpdateScheduledEvent` fields:

| Field      | Type  | Description                          |
| ---------- | ----- | ------------------------------------ |
| `name`     | `str` | Event name set at scheduling time    |
| `event_id` | `int` | Matches the `ScheduledEvent.event_id` |
| `payload`  | `str` | Payload string set at scheduling time |
